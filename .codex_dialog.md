## Turn 8 — codex [2026-07-08 09:39:08]

**Findings**

P1 [openai_agent.py:304](/Users/tianye/AgentCall/src/agentcall/agents/openai_agent.py:304): `_dispatch_tool_call()` sends tool output to whatever `self._ws` is current after `to_thread()` returns. If the old websocket requested the tool, then disconnects and reconnect succeeds while the tool is running, the result is sent to the new session with an old `call_id`. That can produce an OpenAI error or a silent tool response loss, and `_handled_tool_calls` already suppresses retry. Capture the originating websocket/session generation when scheduling the task, and only return the function output if the connection is still the same live session.

P1 [openai_agent.py:299](/Users/tianye/AgentCall/src/agentcall/agents/openai_agent.py:299): tool dispatch exceptions are not contained. If `self._tools.dispatch()` raises, or `json.dumps(result)` fails, the background task dies after marking `call_id` handled, no `function_call_output` is sent, no `response.create` follows, and the model can wait silently forever. This needs a broad exception path that returns an error-shaped tool output or marks the agent fatal.

P1 [openai_agent.py:285](/Users/tianye/AgentCall/src/agentcall/agents/openai_agent.py:285): `error` events are only logged. A rejected `session.update`, invalid tool output, auth/protocol/rate error, or failed response can leave the websocket open while the call loop keeps running with no AI output. At minimum, unrecoverable Realtime `error` events should set `fatal=True`; `response.done` should also inspect failed/incomplete status instead of always treating it as debug noise.

P2 [openai_agent.py:191](/Users/tianye/AgentCall/src/agentcall/agents/openai_agent.py:191): `send_audio()` catches send failures but leaves the known-bad websocket installed and does not trigger reconnect/fatal. If the receive loop does not observe the close promptly, the call can keep dropping every uplink frame with `fatal=False`. Safer behavior is to clear/close the ws and wake the reconnect path, or maintain an explicit disconnected state like qwen.

P2 [openai_agent.py:197](/Users/tianye/AgentCall/src/agentcall/agents/openai_agent.py:197): `say()` is not protected the way `send_audio()` is. During a disconnect/stop window it can raise into callers; the greeting path in `call_agent.py` awaits `agent.say()` directly, so a transient send failure can fail the whole call instead of being handled as a provider disconnect.

**Stop Race**

For the exact qwen P0 lesson, I do not see the same “successful reconnect during stop remains挂回” bug when `_connect()` returns normally: `_reconnect()` rechecks `_running` after `_connect()`, and `stop()` also nulls/closes `self._ws`. The remaining risk is boundedness: `stop()` waits for `_recv_task`, so if `_connect()` or `session.update` send stalls, stop is delayed until that await resolves.

**Test Gaps**

The current tests cover happy reconnect and all-fail fatal, but they would not catch: tool call running across reconnect, tool dispatch raising, server `error` causing silent hang, `response.done` failure status, or `say()` send failure.
